## DEC-001 — Multi-agent architecture with Planner + 3 specialist agents

**Status:** Accepted
**Date:** 2026-07-17
**Owner:** Project Lead

### Decision

KHỚP uses a Planner Agent that decomposes complex operational requests into tasks, then delegates to three specialist agents: Reconciliation Agent, Tax & Compliance Agent, and Merchant Operations Agent. Each agent has its own tool allowlist, output schema, and RAG source.

### Rationale

A single LLM call cannot reliably handle reconciliation, tax validation, and case management in one pass. Separating concerns allows deterministic tools to handle certainty (reference matching, tax formulas) while AI handles ambiguity (Vietnamese text interpretation, candidate scoring, message drafting).

### Affected areas

- `06-system-architecture.md`
- `AI-ADVISOR.md`
- `04-functional-requirements.md`
- `07-data-models.md` (agent_runs, tool_calls tables)

### Rejected alternatives

- **Single-agent with function calling:** Too much context for one agent; loses traceability per specialty.
- **Pipeline without Planner:** Cannot handle variable task ordering or conditional delegation.

### Verification

- Planner produces a plan with ≥3 steps for a complex request
- Each specialist agent calls only tools in its allowlist
- Agent trace shows which agent called which tool

## DEC-002 — Canonical Event Ledger as single source of truth

**Status:** Accepted
**Date:** 2026-07-17
**Owner:** Tech Lead

### Decision

All data sources (SHB transactions, POS, cash sessions, invoices, COD) are normalized into a Canonical Event Ledger in PostgreSQL. Each source goes through an adapter that maps external schemas to the canonical schema.

### Rationale

Without a canonical ledger, reconciliation logic must handle N source formats. With adapters, the core reconciliation engine works against one schema regardless of source.

### Affected areas

- `06-system-architecture.md`
- `07-data-models.md`
- `10-integration.md`
- `ALGORITHM.md`

### Rejected alternatives

- **Direct query per source:** Fragile, duplicated logic, no single audit trail.
- **Data warehouse approach:** Too heavy for MVP; PostgreSQL with proper schema suffices.

### Verification

- All source data appears in canonical tables after adapter run
- Reconciliation queries only reference canonical tables

## DEC-003 — Payment reference as the primary link key

**Status:** Accepted
**Date:** 2026-07-17
**Owner:** Tech Lead

### Decision

A `payment_reference` (e.g., `PAY-A8F21X`) generated by the system is the primary link between sales orders, payment intents, and bank transactions. The three IDs (POS order_id, payment_reference, bank transaction_id) do not need to match each other but are linked through the payment_reference field in the database.

### Rationale

Vietnamese bank transfer notes are unreliable (missing, abbreviated, or wrong). A system-generated reference embedded in a dynamic QR ensures deterministic matching for new transactions. Fuzzy matching is only needed for legacy or external transactions without reference.

### Affected areas

- `07-data-models.md` (payment_intents, payment_allocations)
- `ALGORITHM.md` (matching algorithm)
- `04-functional-requirements.md`

### Rejected alternatives

- **Use bank transaction note as key:** Unreliable; Vietnamese notes are often empty or ambiguous.
- **Use order ID as key:** Not present in bank transaction data.

### Verification

- Transactions with valid reference auto-match ≥95% of the time
- Transactions without reference go through candidate matching

## DEC-004 — Tax Rules Engine is deterministic and separate from LLM

**Status:** Accepted
**Date:** 2026-07-17
**Owner:** Tech Lead

### Decision

The Tax Rules Engine is a deterministic service with versioned rules stored in PostgreSQL. The LLM may explain rules in plain language but may not compute tax formulas, modify rules, or create new tax rates.

### Rationale

Tax calculations must be reproducible and auditable. LLMs are non-deterministic and cannot be relied upon for exact financial calculations. Rule versioning ensures that reports reference the exact rule set in effect at generation time.

### Affected areas

- `COMPLIANCE.md`
- `AI-ADVISOR.md`
- `07-data-models.md` (tax_rule_versions table)
- `04-functional-requirements.md`

### Rejected alternatives

- **LLM for tax calculation:** Non-deterministic, not auditable, legally risky.
- **Hardcoded rules:** Cannot version or update without code changes.

### Verification

- Tax-readiness report includes rule_version and effective_date
- LLM never calls a tax calculation tool; it only calls `retrieve_tax_rules` and `classify_revenue_category`

## DEC-005 — Exception-first UX with human approval for ambiguous cases

**Status:** Accepted
**Date:** 2026-07-17
**Owner:** Project Lead

### Decision

The UI shows only exceptions requiring human decisions, not the full transaction list. AI suggestions include confidence scores. Transactions below the auto-match threshold (≥95%) require human confirmation. The AI never auto-resolves ambiguous transactions.

### Rationale

Banking staff cannot review hundreds of transactions manually. Showing only exceptions reduces workload by 80%+ while maintaining human control over risky decisions.

### Affected areas

- `DESIGN.md` (Exception Inbox screen)
- `04-functional-requirements.md`
- `AI-ADVISOR.md` (confidence thresholds)

### Rejected alternatives

- **Show all transactions:** Overwhelming; defeats the purpose.
- **Auto-resolve everything:** Too risky for financial data; compliance requires human approval.

### Verification

- Exception Inbox shows only items with confidence < 95%
- No transaction is auto-resolved below the threshold
- Human approval is logged in audit_events

## DEC-006 — Next.js + FastAPI + PostgreSQL stack

**Status:** Accepted
**Date:** 2026-07-17
**Owner:** Tech Lead

### Decision

Frontend: Next.js with TypeScript. Backend: FastAPI with Python. Database: PostgreSQL with pgvector. Agent orchestration: LangGraph or custom state machine. Tool protocol: MCP or typed function calling.

### Rationale

Next.js provides SSR and good DX for dashboards. FastAPI is fast, typed, and Python ecosystem supports AI libraries. PostgreSQL with pgvector enables RAG without a separate vector DB. LangGraph provides stateful agent orchestration.

### Affected areas

- `06-system-architecture.md`
- `13-environment-setup.md`
- `IMPLEMENTATION-PLAN.md`

### Rejected alternatives

- **Django:** Heavier; less async-friendly for agent workloads.
- **Separate vector DB (Pinecone):** Adds infrastructure; pgvector suffices for MVP.
- **Node.js backend:** Python has better AI ecosystem support.

### Verification

- Frontend builds with `npm run build`
- Backend starts with `uvicorn main:app`
- pgvector extension enabled in PostgreSQL

## DEC-008 — Audit every agent action

**Status:** Accepted
**Date:** 2026-07-17
**Owner:** Security Lead

### Decision

Every tool call, agent decision, and human approval is recorded in an `audit_events` table with actor, tool, input hash, output hash, rule version, confidence, and timestamp. Audit logs are exportable as JSON or CSV.

### Rationale

Banking operations require full traceability. The audit log is the legal and operational evidence that decisions were made correctly.

### Affected areas

- `11-security.md`
- `07-data-models.md` (audit_events table)
- `COMPLIANCE.md`

### Rejected alternatives

- **Log to files:** Not queryable; harder to export and verify.
- **Application-level logging only:** No structured schema; not auditable.

### Verification

- Every tool call has a corresponding audit_event record
- Audit log export produces valid JSON/CSV

## DEC-009 — Dynamic QR for payment intent

**Status:** Accepted
**Date:** 2026-07-17
**Owner:** Tech Lead

### Decision

Each payment creates a payment intent with a unique reference and a dynamic QR code that pre-fills amount and reference. The QR expires after 15 minutes. Bank webhooks match transactions to orders via the reference.

### Rationale

Static QR codes cause ambiguity when multiple customers pay the same amount simultaneously. Dynamic QR with reference ensures deterministic matching.

### Affected areas

- `04-functional-requirements.md`
- `07-data-models.md` (payment_intents table)
- `10-integration.md` (SePay/SHB webhook)

### Rejected alternatives

- **Static QR:** Ambiguous; cannot distinguish same-amount payments.
- **Manual entry of reference:** Error-prone; customers often skip it.

### Verification

- QR contains amount and reference
- Webhook with reference auto-matches to correct order
